# ATBS 服务拆分原则说明

## 1. 服务拆分背景

ATBS（航空票务预订系统）是一个典型的企业级应用，采用微服务架构进行服务拆分。本项目将单体应用拆分为5个核心微服务，遵循领域驱动设计（DDD）原则和单一职责原则。

## 3. 拆分原则详解

### 3.1 按业务领域拆分（Domain-Driven Design）

| 服务名称 | 业务领域 | 核心职责 | 拆分理由 |
|:--------|:--------|:--------|:---------|
| **Auth Service** | 认证授权域 | 用户认证、OAuth2授权、权限管理、JWT签发 | 认证逻辑独立，可被所有服务复用 |
| **User Service** | 用户域 | 用户管理、乘客管理、员工管理 | 用户数据核心，与其他服务解耦 |
| **Order Service** | 订单域 | 订单管理、票务管理、支付处理 | 订单生命周期独立，事务边界清晰 |
| **Airplane Service** | 航班域 | 航班管理、航线管理、座位预订 | 航班资源管理，与订单服务协作 |
| **Prediction Service** | 预测域 | 客流预测、ARIMA模型训练 | 独立算法服务，Python技术栈 |

### 3.2 按数据独立性拆分

```
数据独立性分析：

Auth Service    ──►  auth_db (用户认证表、角色权限表)
                       │
User Service    ──►  user_db (用户信息表、乘客信息表、员工表)
                       │
Order Service   ──►  order_db (订单表、票务表、支付记录表)
                       │
Airplane Service──►  airplane_db (航班表、航线表、座位预订表)
                       │
Prediction Service──► 模型文件、预测结果（文件存储）
```

**拆分原则：**
1. **数据自治**：每个服务拥有独立的数据库，避免数据耦合
2. **数据隔离**：服务间通过API交互，不直接访问对方数据库
3. **数据一致性**：使用分布式事务（Saga模式）保证最终一致性

### 3.3 按技术栈拆分

| 服务 | 技术栈 | 拆分理由 |
|:-----|:------|:---------|
| Java服务群 | Spring Boot + Spring Cloud | 统一技术栈，便于维护 |
| Prediction Service | Python + FastAPI | 算法模型训练需要Python生态（statsmodels、scikit-learn） |

**技术栈分离的优势：**
1. **专业工具**：Python在数据科学领域有丰富库支持
2. **性能优化**：算法计算使用Python，业务逻辑使用Java
3. **团队分工**：算法团队和业务团队可独立开发

## 4. MVC架构与三层架构映射

### 4.1 传统三层架构 vs 微服务架构

```
传统单体三层架构：
┌─────────────────────────────────────────┐
│              表示层 (Presentation)        │  ← Controller
├─────────────────────────────────────────┤
│              业务逻辑层 (Business)        │  ← Service
├─────────────────────────────────────────┤
│              数据访问层 (Data Access)      │  ← Mapper/DAO
└─────────────────────────────────────────┘

微服务架构（每个服务内部仍保持三层）：
┌─────────────────────────────────────────────────────────────────┐
│  Service A (Auth) │ Service B (User) │ Service C (Order) │ ...  │
├───────────────────┼──────────────────┼───────────────────┼──────┤
│  Controller       │  Controller      │  Controller       │ ...  │
├───────────────────┼──────────────────┼───────────────────┼──────┤
│  Service          │  Service         │  Service          │ ...  │
├───────────────────┼──────────────────┼───────────────────┼──────┤
│  Mapper/DAO       │  Mapper/DAO      │  Mapper/DAO       │ ...  │
└───────────────────┴──────────────────┴───────────────────┴──────┘
```

### 4.2 微服务内部模块划分

每个微服务采用多模块Maven结构，对应三层架构：

```
service-name/
├── service-name-api/           ← 表示层（Controller）
│   └── src/main/java/
│       └── api/v1/
│           ├── app/            # C端接口
│           ├── mgmt/           # 管理端接口
│           ├── internal/       # 内部服务调用
│           └── notify/         # 回调通知
│
├── service-name-biz/           ← 业务逻辑层（Service）
│   └── src/main/java/
│       ├── service/            # 业务接口
│       ├── service/impl/       # 业务实现
│       ├── mapper/             # 数据访问
│       ├── client/             # Feign客户端
│       └── state/              # 状态机
│
├── service-name-bo/            ← 业务对象层（Entity/DTO）
│   └── src/main/java/
│       ├── eo/                 # Entity实体
│       └── dto/                # 数据传输对象
│
├── service-name-ao/            ← 应用对象层（Request/Response）
│   └── src/main/java/
│       ├── req/                # 请求对象
│       └── res/                # 响应对象
│
├── service-name-common/        ← 公共常量/枚举
│   └── src/main/java/
│       ├── enums/              # 枚举
│       └── constants/          # 常量
│
└── service-name-boot/          ← 启动模块
    └── src/main/java/
        └── Application.java    # SpringBoot启动类
```

## 5. 服务间通信设计

### 5.1 同步通信（Feign Client）

```java
// Order Service 调用 User Service
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/api/internal/v1/user/{userId}")
    UserDTO getUserById(@PathVariable Long userId);
}

// Order Service 调用 Airplane Service
@FeignClient(name = "airplane-service")
public interface AirplaneClient {
    @GetMapping("/api/internal/v1/schedule/{scheduleId}")
    ScheduleDTO getScheduleById(@PathVariable Long scheduleId);
}
```

**使用场景：**
- 实时性要求高的查询
- 需要立即返回结果的操作
- 数据一致性要求高的场景

### 5.2 异步通信（消息队列）

```java
// 订单状态变更事件
public class OrderStatusChangedEvent {
    private Long orderId;
    private OrderStatus oldStatus;
    private OrderStatus newStatus;
    private LocalDateTime eventTime;
}

// 发布事件
applicationEventPublisher.publishEvent(new OrderStatusChangedEvent(...));
```

**使用场景：**
- 解耦服务间依赖
- 削峰填谷
- 最终一致性场景

## 6. 服务拆分边界划分

### 6.1 边界划分原则

| 原则 | 说明 | 实践 |
|:-----|:-----|:-----|
| **高内聚** | 相关功能放在同一服务 | 订单和票务在同一服务 |
| **低耦合** | 服务间依赖最小化 | 通过API交互，不直接访问数据库 |
| **单一职责** | 一个服务只负责一个业务领域 | Auth只负责认证 |
| **独立部署** | 每个服务可独立发布 | 服务间版本解耦 |

### 6.2 边界划分示例

```
边界划分决策：

Q: 为什么订单和票务在同一服务？
A: 订单和票务是强关联的，票务的生命周期由订单驱动，
   放在同一服务可避免分布式事务，保证数据一致性。

Q: 为什么用户和乘客分开？
A: 用户（User）是系统账号，乘客（Passenger）是乘机人信息，
   一个用户可以有多个乘客，业务边界清晰。

Q: 为什么航班服务独立？
A: 航班资源是核心业务数据，需要独立管理，
   可被订单服务、预测服务等多个服务调用。

Q: 为什么预测服务独立？
A: 预测服务使用Python技术栈，与Java服务群技术异构，
   且算法模型需要独立部署和扩展。
```

## 7. 服务拆分收益

### 7.1 技术收益

| 收益点 | 说明 |
|:------|:-----|
| **独立扩展** | 根据负载单独扩展某个服务 |
| **技术异构** | 不同服务可使用不同技术栈 |
| **故障隔离** | 单个服务故障不影响整体 |
| **独立部署** | 服务可独立发布，降低风险 |

### 7.2 业务收益

| 收益点 | 说明 |
|:------|:-----|
| **团队自治** | 每个团队负责独立服务 |
| **快速迭代** | 小范围变更快速上线 |
| **业务聚焦** | 服务与业务领域对齐 |
| **复用性** | 认证服务等可被复用 |

## 8. 服务拆分挑战与应对

### 8.1 分布式事务

**挑战：** 跨服务操作的数据一致性

**应对方案：**
1. **Saga模式**：长事务拆分为本地事务序列
2. **TCC模式**：Try-Confirm-Cancel补偿机制
3. **最终一致性**：接受短暂不一致，保证最终一致

### 8.2 服务间通信

**挑战：** 网络延迟、服务雪崩

**应对方案：**
1. **熔断降级**：Hystrix/Resilience4j熔断保护
2. **限流**：Gateway层统一限流
3. **重试机制**：指数退避重试策略

### 8.3 数据一致性

**挑战：** 分布式环境下数据同步

**应对方案：**
1. **事件驱动**：通过消息队列异步同步
2. **CDC机制**：数据库变更捕获
3. **定期对账**：异步补偿修正

## 9. 总结

ATBS的服务拆分遵循以下核心原则：

1. **领域驱动**：按业务领域划分服务边界
2. **数据自治**：每个服务拥有独立数据存储
3. **技术异构**：算法服务使用Python，业务服务使用Java
4. **分层清晰**：每个服务内部保持MVC三层架构
5. **通信规范**：同步用Feign，异步用消息队列

通过合理的服务拆分，ATBS实现了：
- 高内聚、低耦合的服务架构
- 独立部署、独立扩展的能力
- 团队并行开发的效率提升
- 系统整体可用性的提高
