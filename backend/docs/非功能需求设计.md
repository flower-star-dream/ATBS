# ATBS 非功能需求设计

## 1. 性能需求

### 1.1 响应时间指标

| 场景类型 | 指标要求 | 说明 |
|:--------|:--------|:-----|
| **页面加载** | ≤ 2秒 | 首屏加载时间（First Contentful Paint） |
| **API响应** | ≤ 500ms | 95%的接口响应时间（P95） |
| **数据库查询** | ≤ 100ms | 单条查询响应时间 |
| **复杂查询** | ≤ 1秒 | 分页查询、聚合查询 |
| **文件上传** | ≤ 5秒 | 普通文件（<10MB）上传 |
| **预测计算** | ≤ 3秒 | ARIMA模型预测响应 |

### 1.2 吞吐量指标（QPS）

| 服务 | QPS目标 | 峰值QPS | 说明 |
|:-----|:-------:|:-------:|:-----|
| **Gateway** | 5,000 | 10,000 | 统一入口，需要高吞吐 |
| **Auth Service** | 2,000 | 5,000 | 登录、认证接口 |
| **User Service** | 3,000 | 6,000 | 用户信息查询 |
| **Order Service** | 1,000 | 3,000 | 订单创建（涉及事务） |
| **Airplane Service** | 2,000 | 4,000 | 航班查询 |
| **Prediction Service** | 100 | 500 | 预测计算（计算密集型） |

### 1.3 并发用户数

| 场景 | 并发用户数 | 说明 |
|:-----|:----------:|:-----|
| **日常运营** | 500 | 平均在线用户 |
| **促销高峰** | 5,000 | 秒杀/促销活动期间 |
| **系统上限** | 10,000 | 系统设计的最大并发 |

## 2. 可用性需求

### 2.1 可用性指标

| 级别 | 可用性 | 年停机时间 | 适用场景 |
|:-----|:------:|:----------:|:---------|
| **核心服务** | 99.9% | ≤ 8.76小时 | 订单、支付服务 |
| **一般服务** | 99.5% | ≤ 43.8小时 | 用户、航班查询 |
| **非核心服务** | 99% | ≤ 87.6小时 | 预测、统计服务 |

### 2.2 故障恢复指标

| 指标 | 目标值 | 说明 |
|:-----|:------:|:-----|
| **RTO（恢复时间目标）** | ≤ 5分钟 | 服务故障后恢复时间 |
| **RPO（恢复点目标）** | ≤ 1分钟 | 数据丢失容忍度 |
| **故障检测时间** | ≤ 30秒 | 从故障发生到被发现 |
| **自动切换时间** | ≤ 10秒 | 主备切换时间 |

### 2.3 容错设计

```
容错策略：

1. 熔断降级（Circuit Breaker）
   - 失败率达到50%触发熔断
   - 熔断后30秒进入半开状态
   - 提供降级数据（缓存/默认值）

2. 限流保护（Rate Limiting）
   - Gateway层：IP限流 1000次/分钟
   - 服务层：用户限流 100次/分钟
   - 接口层：关键接口 10次/秒

3. 超时控制
   - 服务间调用：3秒超时
   - 数据库查询：5秒超时
   - 外部HTTP调用：10秒超时

4. 重试机制
   - 失败重试3次
   - 指数退避策略：1s, 2s, 4s
   - 仅对幂等操作重试
```

## 4. 安全性需求

### 4.1 认证授权

| 安全项 | 要求 | 实现方式 |
|:------|:----:|:---------|
| **身份认证** | 必须 | JWT Token + OAuth2 |
| **权限控制** | RBAC | 角色-权限-资源模型 |
| **Token过期** | 2小时 | Access Token有效期 |
| **Token刷新** | 7天 | Refresh Token有效期 |
| **密码策略** | 强密码 | 8位以上，含大小写+数字 |

### 4.2 数据安全

| 安全项 | 要求 | 实现方式 |
|:------|:----:|:---------|
| **传输加密** | 必须 | HTTPS/TLS 1.3 |
| **存储加密** | 敏感字段 | AES-256加密 |
| **密码存储** | 必须 | BCrypt哈希 |
| **SQL注入防护** | 必须 | 参数化查询 |
| **XSS防护** | 必须 | 输入过滤+输出编码 |

### 4.3 审计日志

| 审计项 | 记录内容 | 保留时间 |
|:------|:---------|:--------:|
| **登录日志** | 用户、时间、IP、设备 | 1年 |
| **操作日志** | 用户、操作、资源、结果 | 2年 |
| **异常日志** | 异常类型、堆栈、上下文 | 6个月 |
| **访问日志** | URL、参数、响应时间 | 3个月 |

- 平均响应时间

## 10. 总结

ATBS系统的非功能需求设计围绕以下核心目标：

1. **高性能**：确保用户体验，API响应时间控制在500ms内
2. **高可用**：核心服务可用性达到99.9%，故障快速恢复
3. **易扩展**：支持水平扩展，应对业务增长
4. **安全可靠**：多层安全防护，数据加密传输存储
5. **可监控**：全链路监控，及时发现问题
6. **易维护**：自动化部署，完善的备份恢复机制

这些非功能需求将指导系统的架构设计、开发实现和运维保障，确保ATBS系统能够稳定、高效地支撑业务发展。
